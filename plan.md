# План хакатона «Умный реверсивный коридор» — 24–48 часов, 3 разработчика

> **Кейс:** ЦЭТ-2026, школьники. Адаптивное управление реверсивным светофором на ремонтном участке 0.5–2 км. Замена временного управления на систему, которая по данным камер/детекторов гарантирует пустую зону перед сменой фазы и адаптирует длительности под спрос.

---

## 0. Зафиксированные решения

| Тема | Решение |
|---|---|
| Видео для CV | Готовые ролики с YouTube/датасетов (2–3 шт.: пустая дорога, час пик, фура) |
| Алгоритм счёта | YOLOv8n + ByteTrack, виртуальные линии въезда/выезда, счёт по пересечению ID |
| Web-стек | React 18 + TypeScript + Vite + Tailwind |
| Транспорт между сервисами | MQTT (Mosquitto) |
| Транспорт MQTT → браузер | WebSocket-мост (FastAPI), фронт подписан на топики через мост |
| Backend-контроллер | Python 3.11, FastAPI + asyncio + paho-mqtt |
| Хранение | InfluxDB (метрики time-series) + SQLite (события, журнал фаз) |
| Графики | Recharts внутри React (без внешней Grafana) |
| Структура | Monorepo, 3 папки + общий `docker-compose.yml` |
| Время | 24–48 часов, MVP-режим |

**Главная демо-метрика:** сравнение «до/после» по средней задержке, максимальной очереди, пропускной способности и числу аварийных смен фаз. Должно быть видно невооружённым глазом на дашборде.

## 2. MQTT-контракт (фиксируется в первый час)

**Брокер:** `mosquitto:1883` (внутри docker-compose), без auth для хакатона.

### Топики

| Топик | Издатель | Подписчик | Частота |
|---|---|---|---|
| `traffic/sensor/{side}/entry` | CV | Control | по событию |
| `traffic/sensor/{side}/exit` | CV | Control | по событию |
| `traffic/sensor/midzone` | CV | Control | каждые 1с |
| `traffic/state/phase` | Control | Web (через ws), CV | при смене + раз в 1с |
| `traffic/state/queues` | Control | Web | каждые 500мс |
| `traffic/state/metrics` | Control | Web | каждые 1с |
| `traffic/event/log` | Control | Web | по событию |
| `traffic/cmd/override` | Web | Control | по нажатию |
| `traffic/cmd/scenario` | Web | Control + CV | при запуске сценария |

`{side}` ∈ `{A, B}`.

### Схемы payload (JSON, всё в `contracts/events.schema.json`)

```jsonc
// traffic/sensor/{side}/entry
{
  "ts": 1716000000.123,        // unix seconds, float
  "side": "A",                  // "A" | "B"
  "vehicle_id": "trk_42",       // ByteTrack ID
  "vehicle_class": "car",       // "car" | "truck" | "bus" | "emergency"
  "confidence": 0.91
}

// traffic/sensor/{side}/exit  — тот же формат

// traffic/sensor/midzone
{ "ts": ..., "vehicles_in_zone": ["trk_42", "trk_43"], "stuck_ids": [] }

// traffic/state/phase
{
  "ts": ...,
  "phase": "GREEN_A",           // GREEN_A | GREEN_B | ALL_RED | EMERGENCY
  "phase_started_at": ...,
  "phase_will_end_at": ...,     // null если адаптивно
  "reason": "queue_pressure"    // human-readable
}

// traffic/state/queues
{ "ts": ..., "queue_A": 7, "queue_B": 0, "wait_A_sec": 12, "wait_B_sec": 0 }

// traffic/state/metrics
{
  "ts": ...,
  "avg_delay_A": 18.4, "avg_delay_B": 5.1,
  "max_queue_A": 12, "max_queue_B": 3,
  "throughput_per_hour": 380,
  "unsafe_switches": 0,
  "mode": "adaptive"            // "fixed" | "adaptive"
}

// traffic/event/log
{ "ts": ..., "level": "warn", "code": "STUCK_VEHICLE", "msg": "trk_99 inside >30s", "data": {...} }

// traffic/cmd/override
{ "action": "FORCE_GREEN_A" | "FORCE_GREEN_B" | "ALL_RED", "operator": "demo" }

// traffic/cmd/scenario
{ "scenario_id": "rush_hour" | "asymmetric" | "stuck_truck" | "emergency", "mode": "fixed" | "adaptive" }
```

**Правило:** любое изменение схемы — PR в `contracts/`, приостановка работы остальных, согласование за 5 минут в чате.

## 3. Сценарии демонстрации (фиксируем заранее)

В `contracts/scenarios.md`. CV должен уметь воспроизводить эти видео, Control — адекватно реагировать, Web — показывать «до/после».

| ID | Видео/режим | Что демонстрирует |
|---|---|---|
| `baseline` | Любое видео, фиксированные фазы 3+3 мин | Базовая линия для сравнения |
| `rush_hour` | Поток сильно асимметричный | Адаптив сокращает зелёное на пустой стороне |
| `stuck_truck` | Фура медленно проезжает мимо линии выезда | Контроллер ждёт `count_in == count_out`, не переключает преждевременно |
| `emergency` | Скорая на стороне B | Ручной override, `ALL_RED` → `GREEN_B` |
| `comm_loss` | Принудительный обрыв midzone-сообщений | Резервный таймер, событие в журнал |

Для каждого сценария Web показывает на одном экране сравнение метрик `fixed` vs `adaptive`.

## 4. Распределение по разработчикам

### Вайбкодер A — Computer Vision (`services/cv/`)

**Цель:** превращает входное видео в поток MQTT-событий «машина пересекла линию въезда/выезда» и «машины внутри зоны».

**Задачи:**
1. Подобрать 3 видео под сценарии (`baseline`, `rush_hour`, `stuck_truck`). Положить в `services/cv/videos/`. **Не коммитить в git** — только ссылки в README.
2. Сделать утилиту `configs/zone_picker.py`: открывает первый кадр, мышью кликаешь 4 точки → линия въезда A, линия выезда A, линия въезда B, линия выезда B + полигон зоны. Сохраняет в `configs/{video_name}.json`.
3. Запустить YOLOv8n (ultralytics) + ByteTrack на видео, классы `car|truck|bus`.
4. Для каждого track ID отслеживать пересечение линий → публиковать `entry`/`exit`.
5. Раз в секунду публиковать `midzone`: какие track ID находятся внутри полигона зоны, и какие из них стоят на месте >30с (через скорость bbox).
6. Распознавание «emergency» — на хакатоне честно: помечаем заранее выбранную track ID как emergency через CLI-флаг (нет смысла обучать).
7. CLI: `python -m cv_service --video videos/rush.mp4 --config configs/rush.json --speed 4x`.
8. Поддержать `traffic/cmd/scenario` — запускать соответствующее видео.

**DoD:**
- На любом из 3 видео при просмотре отображается окно с bbox + линиями + счётчиками (для отладки).
- В `mosquitto_sub -t 'traffic/sensor/#' -v` видны корректные события.
- За 60 секунд видео `rush_hour` события `entry_A` и `exit_A` совпадают по числу с реальностью ±2 машины.
- Готов к 16-му часу хакатона.

**Стек:** `ultralytics`, `supervision` (для ByteTrack и аннотаций), `opencv-python`, `paho-mqtt`, `pydantic`.

**Риски:**
- ByteTrack теряет ID на закрытиях — закладываем толерантность ±2 машины и таймаут принудительной чистки в Control.
- На медленной машине YOLO не успевает realtime — отдаём флаг `--speed 1x` (для демо ставим 0.5x видео, чтобы выглядело красиво).

### Вайбкодер B — Control / Backend (`services/control/`)

**Цель:** мозг проекта. Получает события CV, рулит фазами по политике, гарантирует безопасность, отдаёт состояние во фронт через WebSocket.

**Задачи:**
1. **FSM фаз** (`fsm.py`): состояния `GREEN_A`, `ALL_RED_A_to_B`, `GREEN_B`, `ALL_RED_B_to_A`, `EMERGENCY`, `MANUAL`.
2. **Безопасность** (`safety.py`):
   - Считает `count_in[A]`, `count_out[B]` по событиям из MQTT.
   - Смена с `GREEN_A` на `GREEN_B` разрешена только при `count_in[A] == count_out[B]` И `vehicles_in_zone == 0`.
   - Жёсткий верхний лимит ожидания: расчётное время проезда последней машины + 60с. Если истекло — событие `STUCK_VEHICLE`, вызов оператора, фаза заморожена.
3. **Адаптивная политика** (`policy.py`):
   - `T_min` = время на проезд 3 машин (≈ 15с), `T_max` = 5 мин.
   - Приоритет = `w1 * длина_очереди + w2 * время_ожидания_красного`. Стартовые `w1=1, w2=0.1`.
   - Если очередь на стороне = 0 более 30с → принудительно снимаем зелёное в пользу другой стороны.
4. **Режимы:**
   - `fixed` — классические 3+3 минуты, для baseline.
   - `adaptive` — основная политика.
   - Переключение через `traffic/cmd/scenario` — для прямого сравнения «до/после».
5. **Override:** обработка `traffic/cmd/override` — `FORCE_GREEN_A/B` (для скорой), `ALL_RED` (паника).
6. **Резервный режим:** если за 5с нет ни одного `midzone`-сообщения → переход в безопасный таймер 4+4 мин, событие в журнал.
7. **WebSocket-мост** (`ws_gateway.py`): подписан на все `traffic/state/#` и `traffic/event/#`, форвардит на `/ws` всем подключённым клиентам.
8. **REST** (`api.py`):
   - `POST /scenario/start` — публикует `traffic/cmd/scenario`.
   - `POST /override` — публикует `traffic/cmd/override`.
   - `GET /metrics/compare?scenario=rush_hour` — отдаёт две сессии (fixed + adaptive) для сравнения.
9. **Хранение:**
   - Influx — все потоковые метрики (queues, throughput).
   - SQLite — журнал событий (`event_log`), журнал фаз (`phase_log`), сессии сценариев (`session`).
10. **Тесты** (юнит на FSM и safety): обязательно. Без них демо развалится.

**DoD:**
- `pytest` зелёный, ≥10 тестов на FSM/safety.
- Скрипт `scripts/replay.py` подаёт фейковые MQTT-события — Control не делает unsafe-переключение ни разу за 1000 итераций.
- Готов к 24-му часу.

**Стек:** `fastapi`, `uvicorn`, `paho-mqtt`, `pydantic`, `aiosqlite`, `influxdb-client`, `pytest`, `pytest-asyncio`.

### Вайбкодер C — Web Dashboard (`web/`)

**Цель:** один экран, на котором всё видно: схема дороги, фазы, очереди, метрики «до/после», журнал, кнопки оператора.

**Задачи:**
1. Скелет: Vite + React + TS + Tailwind + Recharts. Подключение к WebSocket контроллера через `useWebSocket`.
2. **Схема дороги** (`RoadCanvas.tsx`) — HTML canvas:
   - Двухполосная дорога, ремонтная зона посередине.
   - Светофоры на концах, цвет по `traffic/state/phase`.
   - Машинки по событиям `entry`/`exit` плывут вдоль зоны (упрощённая анимация, не точная физика).
   - Очередь на въезде — стопка прямоугольников, длина = `queue_A/B`.
   - Подсветка «застрявшего» ТС из `stuck_ids`.
3. **Phase timeline** (`PhaseTimeline.tsx`): горизонтальная лента последних 5 минут — кто был зелёный, когда переключения.
4. **Метрики «до/после»** (`MetricsPanel.tsx`):
   - 4 KPI-карточки: avg_delay, max_queue, throughput, unsafe_switches.
   - Каждая карточка показывает значения для `fixed` и `adaptive`, с дельтой и стрелкой.
   - Графики Recharts: throughput за время, длина очереди за время.
5. **Журнал** (`EventLog.tsx`): последние 50 событий с цветом по уровню.
6. **Управление**:
   - Селектор сценария (`baseline`, `rush_hour`, `stuck_truck`, `emergency`) → `POST /scenario/start`.
   - Тумблер `fixed`/`adaptive` (запускает ту же сценарку в другом режиме для сравнения).
   - Кнопки `Force Green A/B`, `All Red`, `Emergency` → `POST /override`.
7. **Презентационный режим:** клавиша `F` — fullscreen, скрыть управление, оставить дашборд (для финального показа).
8. Типы строго из `contracts/events.schema.json` (через `json-schema-to-typescript`).

**DoD:**
- На демо-машине запускается одной командой и реагирует на события за <500мс.
- KPI-карточки наглядно показывают разницу между `fixed` и `adaptive` (минимум 30% улучшения по avg_delay в `rush_hour`).
- Готов к 36-му часу — оставляем 8–12 часов на полировку и интеграцию.

**Стек:** `react`, `vite`, `typescript`, `tailwindcss`, `recharts`, `zustand` (для глобального стейта), `lucide-react` (иконки).

## 7. Вопросы, которые остались для команды (решить в час 0–1)

1. **Кто Артём (тимлид)?** Если в команде ровно три разработчика — один из них берёт роль интегратора по совместительству. Тогда у него меньше «ядра», больше клея.
2. **Один или два дашборда?** В плане один экран с переключением fixed/adaptive. Альтернатива — два рядом (split-view) для одновременного сравнения. Решить, когда C сделает скелет.
3. **Звук на демо?** Если будет проектор с колонками — у светофора можно добавить «бип» при смене фазы. Дёшево, эффектно. Опционально.
4. **OCR номеров?** В кейсе упомянут EasyOCR/Tesseract. На MVP **не делаем** — приоритеты: безопасность фаз и метрики. Если останется время после часа 20 — A может попробовать как «вишенку».
5. **Питч-слайд внутри дашборда?** Иногда жюри не даёт слайды — хорошо иметь кнопку «О проекте» прямо в Web.

## 8. Что мы НЕ делаем (явные out-of-scope)

- Реальное железо (петли, контроллеры). Только эмуляция.
- ML-обучение под этот кейс. Берём готовый YOLOv8n.
- Multi-camera fusion. Одна камера на сторону достаточно.
- Аутентификация в дашборде.
- Тесты на фронте (только smoke).
- Mobile-вёрстка. Демо на ноутбуке.

## 9. Контракт «не сломай соседа»

1. `contracts/` — лочим в час 1. Любое изменение → PR + апрув всех троих.
2. **Ветки:** `cv/*`, `control/*`, `web/*`. Один разработчик пушит только в свою префикс-ветку.
3. Перед пушем — `docker compose up` локально, проверить что свой сервис стартует.
4. **Не править** `docker-compose.yml` без согласования (можно добавить свой service, не трогая чужие).
5. Если CV отвалился — Control работает с replay.py. Если Control упал — Web показывает «связь потеряна», но не падает.
6. **Замок при коммите:** перед merge в main — `make smoke` (минимальный e2e через replay.py).

## 10. Питч-нарратив (черновик на финал)

> «Реверсивные участки на ремонте дорог сегодня работают по таймеру: 3 минуты туда, 3 обратно. Это даёт два класса проблем — простой одной полосы при асимметричном потоке и риск ДТП, когда фура не успела выехать из зоны. Мы заменили таймер на адаптивную систему, которая видит каждую машину камерой, гарантирует пустую зону перед сменой фазы и распределяет зелёное по реальному спросу. На наших сценариях средняя задержка падает на 30–50%, а число небезопасных смен фазы — до нуля.»

Цифры подставляем после прогона. Главное — чтобы дашборд эти цифры рисовал сам.
